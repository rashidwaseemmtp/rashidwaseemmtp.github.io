05 / System design

How I think about systems.

A few representative architectures from the work above — simplified to show shape and intent, not internal details.

Multi-tenant request flow

Tenant resolution happens at the edge so every downstream layer is tenant-scoped by default — impossible to query across tenants by accident.

Client
   │   subdomain / JWT
   ▼
Edge Router ─► resolve tenant
   │
   ▼
App Layer  (tenant middleware)
   │       ├─ feature flags / plan
   │       ├─ rate limit (per tenant)
   │       └─ scoped queries only
   ▼
Tenant DB schema + Redis (tenant-prefixed)

Subscription state machine

Stripe is the source of truth. Local state is a projection — rebuildable from event log, idempotent on replay.

Stripe ──webhook──► Receiver
                       │  verify + dedupe
                       ▼
                  Event Log (append-only)
                       │
                       ▼
                  Reducer ─► subscription state
                       │
                       ├─► billing_events table
                       └─► outbox ─► email / CRM / ops

Read-heavy API caching

Hot read paths are served from Redis with explicit invalidation on writes — cache is treated as a first-class part of the data layer, not an afterthought.

GET /resource
   │
   ▼
Cache ──hit──► return
   │ miss
   ▼
DB  (with read replica)
   │
   ▼
warm cache ─► return

POST/PUT ─► write DB ─► invalidate keys
01
Boring on purpose

Pick the well-understood option. Save complexity for problems that actually need it.

02
Observable by default

Logs, metrics, and traces aren’t bolted on later — they ship with the feature.

03
Idempotent & replayable

Webhooks, jobs, and integrations should survive retries and partial failures.

04
Tenant isolation as an invariant

Cross-tenant access should be impossible by construction, not by code review.
